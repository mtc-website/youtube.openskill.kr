보따리 청소에는 두 개의 층이 있다

컨텍스트의 'GC'
컨텍스트 정리는 GC가 아니라 손실 요약/제거다.
← 이전목차다음 →

LEARN

이 강을 보고 나면
  • 컨텍스트 정리가 프로그래밍의 GC(가비지 컬렉션)와 무엇이 결정적으로 다른지 — 자동 회수가 아니라 '판단이 들어간 손실 요약'임을 설명할 수 있다.
  • 정리가 일어나는 두 개의 층 — 추론 레이어(KV eviction / StreamingLLM)와 앱 레이어(compaction, tool result clearing)를 구분할 수 있다.
  • compaction이 무엇을 보존하고 무엇을 버리는지, 그리고 '과한 압축'이 왜 위험한지(손실의 위험)를 말할 수 있다.

HOOK

도입

긴 작업을 시키다 보면 어느 순간 보따리(컨텍스트 윈도우)가 가득 찬다. 코드 리팩토링 30분째, 도구 호출 결과가 수십 개 쌓이고, 대화는 길어질 대로 길어졌다. 더 넣을 자리가 없다. 이제 무언가는 버려야 한다.

여기서 많은 사람이 직관적으로 떠올리는 말이 있다. "메모리 관리하듯 가비지 컬렉션 돌리면 되잖아?" 자바나 파이썬을 써 본 사람이라면 익숙한 발상이다. 쓰지 않는 건 알아서 사라지게 두면 된다는 것. 하지만 컨텍스트 청소는 그렇게 깔끔하지 않다. 무엇을 버릴지 자동으로 판별해 줄 장치가 없기 때문이다.

그래서 진짜 질문은 이것이다. 무엇을 버리고 무엇을 남길 것인가, 그리고 그 판단은 누가 하는가? 이번 강에서는 컨텍스트 청소가 GC가 아니라 '손실을 동반한 요약'이라는 점, 그리고 그 청소가 두 개의 서로 다른 층에서 일어난다는 점을 짚는다.

PRINCIPLE

핵심 원리

GC는 도달 가능성으로 죽은 객체를 자동 회수한다 — 컨텍스트엔 그게 없다

프로그래밍 언어의 가비지 컬렉터는 명확한 규칙으로 작동한다. 어떤 객체가 살아 있는 다른 객체로부터 '도달 가능(reachability)'한지를 추적해, 아무도 더 이상 가리키지 않는 객체를 죽은 것으로 판정하고 메모리를 회수한다. 이 판정은 기계적이고 무손실이다. 살아 있는 데이터는 절대 실수로 지워지지 않는다.

컨텍스트에는 이런 자동 판별 장치가 없다. 토큰 더미를 보고 "이 문장은 더 이상 아무도 참조하지 않으니 죽었다"고 기계적으로 판정할 방법이 없다. 무엇이 중요한지는 모델이 '판단'해서 요약해야 한다. 그리고 요약하는 순간 원본의 일부는 사라진다. 즉 컨텍스트 정리는 본질적으로 손실적(lossy)이다.

청소는 두 개의 층에서 일어난다

추론 레이어(inference layer). 모델이 토큰을 처리하는 낮은 수준에서 일어나는 정리다. 대표적으로 KV 캐시 eviction과 StreamingLLM 방식이 있다. StreamingLLM은 맨 앞쪽의 'attention sink' 토큰과 최근 윈도우를 남기고 그 사이 중간 부분을 폐기해, 무한히 긴 스트림에서도 추론이 굴러가게 한다. 이 층은 의미를 이해하고 버리는 게 아니라, 어텐션 구조상 덜 필요한 위치를 폐기하는 데 가깝다.

앱 레이어(application layer). 우리가 흔히 '컨텍스트 관리'라 부르는 높은 수준의 정리다. compaction은 길어진 대화를 요약 압축해 새 윈도우로 재시작하는 것이고, tool result clearing은 이미 다 쓴 도구 호출 결과 같은 '죽은 데이터'를 제거하는 것이다. 이 층은 의미를 판단해 보존·폐기를 결정한다.

compaction은 무엇을 남기고 무엇을 버리나

잘 설계된 compaction은 아키텍처 결정, 아직 해결되지 않은 버그, 핵심 구현 같은 '계속 필요한 것'은 보존하고, 중복된 도구 출력처럼 '이미 소비된 것'은 버린다. 문제는 압축이 과하면 나중에 중요해질 정보를 미리 잃어버린다는 점이다. 이것이 손실의 위험이다 — 지금은 안 쓰는 것 같아 버렸는데, 세 턴 뒤에 바로 그게 필요해지는 상황.

핵심

GC는 도달 가능성으로 죽은 객체를 자동·무손실로 회수한다. 컨텍스트에는 그런 자동 판별이 없어, 무엇이 중요한지 모델이 판단해 요약해야 하고, 그 과정은 본질적으로 손실적이다. 그래서 컨텍스트 청소는 GC가 아니다.

ANALOGY

비유로 이해하기
비유

책상 정리를 떠올려 보자. 메모리의 GC는 자동 청소 로봇이다. 바닥에 떨어진 게 쓰레기인지 아닌지를 센서로 정확히 인식해, 쓰레기만 빨아들이고 멀쩡한 물건은 절대 건드리지 않는다. 판단은 기계적이고, 실수로 중요한 걸 버리는 일이 없다.

컨텍스트 정리는 다르다. 그건 쌓인 서류를 사람이 직접 요약해 버리는 일에 가깝다. "이 회의록은 한 줄로 줄여도 되겠지" "이 영수증은 이제 필요 없겠지" — 매번 판단이 들어가고, 요약하는 순간 원본 디테일은 사라진다. 가끔은 버린 영수증이 한 달 뒤 정산 때 꼭 필요해진다. 자동 로봇과 달리, 손이 가고 실수도 동반하는 작업인 것이다.

MYTHS

흔한 오개념 부수기
흔히 이렇게 말한다정확히는
"컨텍스트는 GC를 못 한다."정리는 한다. 다만 도달 가능성 기반 자동 회수가 아니라, 모델의 판단이 들어간 손실 요약/제거 기반으로 한다.
"요약은 무손실이다 — 핵심만 잘 추리면 된다."요약은 본질적으로 손실적이다. 과하게 압축하면 지금은 안 보여도 나중에 중요해질 정보가 사라진다.
"컨텍스트 정리는 한 가지 방법만 있다."두 층에서 일어난다. 추론 레이어(KV eviction / StreamingLLM)와 앱 레이어(compaction, tool result clearing).

EXPERIMENT

생각 실험
해보기

같은 긴 대화 하나를 준비한다 — 예컨대 한 시간짜리 디버깅 세션 로그. 이걸 compaction 전과 후로 나란히 펼쳐 두고, 다음을 표시해 본다.

① 보존된 것: 어떤 아키텍처 결정이, 어떤 미해결 버그가, 어떤 핵심 코드가 압축본에 살아남았나? ② 사라진 것: 어떤 중간 시도, 어떤 도구 출력, 어떤 대화의 결이 통째로 빠졌나? ③ 위험 표시: 사라진 것 중에서 "어, 이건 나중에 필요할 수도 있는데" 싶은 항목에 빨간 표를 단다.

빨간 표가 늘어날수록, 그 압축은 효율적인 동시에 위험하다. 이 긴장이 바로 손실 요약의 본질이다.

SIMULATOR

시뮬레이터 설계
화면

가운데에 '보따리'가 그려져 있고, 그 안에 항목들이 카드로 떠 있다. 각 카드는 색으로 분류된다 — 초록(핵심 결정·미해결 버그), 회색(중복 도구 출력), 노랑(애매한 중간 시도).

사용자가 [정리] 버튼을 누르면 두 가지 애니메이션이 나뉘어 재생된다. tool result clearing: 회색 카드가 스르륵 빠져나가 비워진다. compaction: 여러 카드가 한 장의 작은 카드로 접혀 합쳐지며 압축된다 — 이때 노랑 카드의 디테일이 살짝 흐려지며 일부 사라지는 효과로 '손실'을 시각화한다.

화면 한쪽엔 토글로 '추론 레이어 보기'가 있어, StreamingLLM처럼 맨 앞 카드(attention sink)와 최근 카드만 또렷이 남고 중간이 폐기되는 모습을 따로 보여 준다. 두 층이 서로 다른 청소임을 한눈에 대비시킨다.

RECAP

한 줄 요약 & 다음 편

컨텍스트 청소는 GC가 아니다. 자동·무손실 회수가 아니라, 모델이 무엇이 중요한지 판단해 요약·제거하는 손실적 작업이며, 추론 레이어(KV eviction / StreamingLLM)와 앱 레이어(compaction, tool result clearing) 두 층에서 일어난다.

그렇다면 손실이 두려운 정보는 어떻게 지킬까? 다음 편 — 버리지 말고 '빼두기'. 보따리 밖에 보관했다 다시 부르는 전략을 다룬다.

NOTES

강의 노트 — 판서 순서
  1. 메모리 GC 정의부터 — 도달 가능성(reachability)으로 죽은 객체를 자동·무손실 회수.
  2. 대비: 컨텍스트엔 자동 판별이 없다 → 모델이 판단해 요약 → 본질적으로 손실적.
  3. 두 층 그리기: 추론 레이어(KV eviction / StreamingLLM: sink + 최근 윈도우, 중간 폐기) vs 앱 레이어(compaction, tool result clearing).
  4. compaction이 보존하는 것(아키텍처 결정·미해결 버그·핵심 구현) vs 버리는 것(중복 도구 출력).
  5. 과한 압축의 위험 강조 — 나중에 중요해질 정보 유실(손실의 위험).
  6. 다음 편 예고: 버리지 말고 빼두기(외부 메모로 잇기).
