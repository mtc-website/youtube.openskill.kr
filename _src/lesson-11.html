도구와 MCP

모델이 세계와 연결되는 법
모델은 도구로만 바깥에 행동·조회한다. MCP는 그 도구를 꽂는 표준이다.
← 이전목차다음 →

LEARN

이 강을 보고 나면
  • 도구(함수 호출)가 실제로 어떻게 동작하는지, 그 '계약'이 무엇인지 설명할 수 있다.
  • 좋은 도구 설계 원칙 — 적고, 명확하고, 토큰 효율적이며, 겹치지 않게 — 을 안다.
  • MCP가 무엇이고 왜 필요한지, 'USB-C 비유'로 누구에게나 설명할 수 있다.

HOOK

도입

이상한 사실 하나. LLM은 인터넷을 직접 보지 못합니다. 데이터베이스에 접속하지도, 파일을 열지도 못합니다. 모델은 본질적으로 텍스트를 받아 텍스트를 내놓는 함수일 뿐이니까요. 그런데 우리는 에이전트가 검색하고, 코드를 실행하고, 메일을 보내는 걸 매일 봅니다.

모델이 직접 못 하는데, 그 일은 어떻게 일어나는 걸까요? 답은 도구(tool)입니다. 모델은 "이 도구를 이렇게 써줘"라고 요청하고, 바깥의 런타임이 실제로 실행한 뒤 결과를 모델에게 돌려줍니다. 모델의 손발은 도구인 셈이죠.

이번 강은 그 손발의 작동 원리, 좋은 손발을 설계하는 법, 그리고 그 손발을 표준 규격으로 꽂는 MCP를 다룹니다.

PRINCIPLE

핵심 원리

(a) 도구 사용(tool use / function calling)

구조는 이렇습니다. 모델은 "검색 도구를 query="..." 인자로 호출하겠다"는 구조화된 출력(보통 JSON)을 냅니다. 이건 아직 행동이 아니라 요청입니다. 그러면 모델 바깥의 런타임이 그 JSON을 받아 실제 함수를 실행하고, 결과를 다시 모델의 컨텍스트(보따리)에 넣어줍니다. 모델은 그 결과를 읽고 다음을 정합니다.

핵심 — 도구의 작동

모델은 "이 도구를, 이 인자로 호출하라"는 구조화된 출력(JSON)을 낸다. 직접 실행하지 않는다.

실제 실행은 런타임이 하고, 그 결과를 모델의 컨텍스트에 다시 넣어준다. 모델의 손발 = 도구.

(b) 도구의 '계약', 그리고 좋은 도구

모델이 도구를 옳게 쓰려면 계약(contract)이 명확해야 합니다. 도구의 이름, 받을 파라미터, 돌려줄 반환값 — 이 세 가지가 또렷할수록 모델은 헷갈리지 않습니다. 모델은 이 계약 설명만 보고 언제 무엇을 쓸지 판단하기 때문이죠.

좋은 도구는 이렇습니다. 적고, 명확하고, 토큰 효율적이며, 기능이 서로 겹치지 않습니다. 핵심 직관 하나 — 사람이 봐도 어떤 도구를 써야 할지 못 정하는 모호한 도구 묶음이라면, 모델도 못 정합니다. 그래서 도구는 많다고 좋은 게 아니라, 많고 모호하면 오히려 해롭습니다.

(c) MCP — 도구를 꽂는 개방 표준

MCP(Model Context Protocol)는 도구와 데이터 소스를 모델/클라이언트에 꽂기 위한 개방 표준입니다. 비유하자면 "도구·데이터의 USB-C"죠. 예전엔 기기마다 충전 단자가 제각각이었듯, 도구를 모델마다 다른 방식으로 붙여야 했습니다. MCP는 그 단자를 하나로 통일합니다.

핵심 — MCP

MCP = 도구·데이터 소스를 모델/클라이언트에 꽂는 개방 표준. "도구·데이터의 USB-C."

한 번 MCP 서버를 만들면, 여러 클라이언트가 같은 방식으로 그 도구를 쓸 수 있다. 표준이라 재사용된다.

ANALOGY

비유로 이해하기
비유

RPG 캐릭터를 생각해 봅시다. 캐릭터(모델)는 맨손으로는 멀리 있는 적도, 닫힌 문도 어쩌지 못합니다. 대신 장비를 쥐면 활로 먼 적을 맞히고, 포탈을 열면 다른 지역에 닿습니다. 도구가 바로 이 장비와 포탈입니다 — 캐릭터가 세계와 상호작용하는 손발이죠.

그런데 장비마다 끼우는 슬롯 규격이 제각각이면 매번 새로 맞춰야 합니다. MCP는 그 장비를 꽂는 표준 단자입니다. 누가 만든 장비든 이 규격만 따르면 어떤 캐릭터에게도 끼울 수 있죠. 만든 사람은 한 번만 만들고, 여러 캐릭터가 같은 방식으로 씁니다.

MYTHS

흔한 오개념 부수기
흔히 이렇게 말한다정확히는
"모델이 DB나 웹을 직접 본다"모델은 텍스트만 다룬다. 외부에 닿는 일은 도구를 통해서만 일어난다. 모델은 '호출 요청'을 낼 뿐, 실행은 런타임이 한다.
"도구는 많을수록 좋다"많고 모호하면 오히려 해롭다. 사람도 못 고를 도구 묶음이면 모델도 못 고른다. 적고·명확한 게 좋다.
"MCP는 특정 회사 기능이다"MCP는 개방 표준이다. 한 번 만든 MCP 서버를 여러 클라이언트가 같은 방식으로 쓴다.

EXPERIMENT

생각 실험
해보기

같은 질문 — 예컨대 "지금 이 도시의 기온은?" — 을 두 조건으로 던져봅시다.

① 도구 없이: 모델은 기억에 의존해 그럴듯한 숫자를 말합니다. 하지만 실제 현재값을 알 길이 없으니 헛소리(환각)가 섞일 위험이 큽니다.

② 도구로: 모델이 날씨 조회 도구를 city="..."로 호출하고, 런타임이 실측값을 가져와 컨텍스트에 넣어줍니다. 모델은 그 실측을 근거로 답합니다.

두 답을 비교해 보면, 도구가 '아는 척'을 '실제로 확인함'으로 바꾼다는 걸 체감하게 됩니다. 모델의 똑똑함이 아니라, 세계에 닿는 통로가 답의 신뢰도를 만든다는 점.

SIMULATOR

시뮬레이터 설계
화면

왼쪽에 모델 박스, 오른쪽에 세계(웹·DB·파일). 둘 사이엔 처음엔 벽이 있습니다. '도구 없음' 모드에서 질문을 던지면 모델이 혼자 답하고, 답 옆에 ⚠ 확인 불가 배지가 붙습니다.

'도구 있음' 토글을 켜면 둘 사이에 도구 슬롯이 생깁니다. 모델이 JSON 호출 카드를 내면 → 런타임이 그 카드를 들고 세계로 갔다가 → 결과를 들고 돌아와 컨텍스트에 꽂습니다. 이제 답 옆엔 ✓ 실측 근거 배지가 붙습니다.

하단엔 MCP 패널: 외부 도구(예: 캘린더, 검색)를 'USB-C 꽂듯' 슬롯에 드래그해 끼우면, 같은 인터페이스로 즉시 사용 가능해지는 모습을 보여줍니다. 도구를 너무 많이/모호하게 꽂으면 모델이 헷갈려 잘못 고르는 시나리오도 한 버튼으로 시연합니다.

RECAP

한 줄 요약 & 다음 편

모델은 도구로만 세계와 닿는다 — 직접 보지 않고 호출을 요청할 뿐이며, 실행은 런타임이 한다. 그 도구를 꽂는 개방 표준이 MCP이고, 좋은 도구는 적고·명확하고·겹치지 않는다.

다음 편 — 일이 커지면 한 캐릭터로 다 못 합니다. 일을 나눠 맡기는 서브에이전트로 갑니다.

NOTES

강의 노트 — 판서 순서
  1. 왼쪽 '모델', 오른쪽 '세계' 그리고 사이에 벽. "모델은 직접 못 본다" 적기.
  2. 벽에 문 하나 그리고 '도구'라 쓰기. 화살표: 모델→JSON 호출→런타임 실행→결과 복귀.
  3. 도구 계약 세 칸: 이름 · 파라미터 · 반환. "명확해야 옳게 쓴다."
  4. 좋은 도구 4원칙: 적게 · 명확 · 토큰 효율 · 안 겹침. "사람도 못 고르면 모델도 못 고른다."
  5. MCP 적고 USB-C 그림. "도구·데이터의 표준 단자, 개방 표준, 한 번 만들면 여러 클라이언트가 재사용."
  6. 마무리: "도구 = 손발, MCP = 표준 단자." 밑줄.
