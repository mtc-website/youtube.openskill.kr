메타코딩 "에이전트 게임 — 컨텍스트를 설계하는 법" 시리즈 기획서 (v2 전체 재설계)

작성 기준일: 2026-05-22 컨셉: 게임 세계를 시리즈 전체를 관통하는 척추(spine)로 삼는다. 매 편이 같은 게임 세계의 다른 규칙을 가르치며, 도구 클릭이 아니라 원리로 무장한다. 타깃: AI 도구는 써봤지만 원리는 모르는 주니어~중급 개발자.

0. 핵심 컨셉 한 줄

"당신은 플레이어다. AI는 기억 없는 캐릭터다. 매 턴 '컨텍스트 보따리'를 통째로 건네야 그가 움직인다 — 그 보따리를 설계하는 법을 배운다."

게임 장르는 롤토체스(TFT) / 파티 RPG 형 으로 통일한다. 이유: 플레이어가 여러 유닛을 부리고(멀티에이전트), 전투는 유닛이 자율 진행하며(루프 도는 에이전트), 아이템은 공유·장착 가능(스킬). 롤(MOBA)은 "1인 1챔피언"이라 멀티에이전트와 안 맞으니 메인 프레임으로 쓰지 않는다. (애니→티버 같은 "소환" 예시는 8편에서 보조 비유로만 인용)

1. 시리즈 메타포 사전 (★ 모든 대본의 기준표)

| 게임 요소 | 에이전트 개념 | 정확도 / 주의 | |---|---|---| | 플레이어(나) | main-agent / 오케스트레이터 | 직접 전투 안 함, 무엇을 할지 결정·배치. | | 유닛/캐릭터 | sub-agent | 각자 독립된 시야(=컨텍스트 윈도우). 나누는 목적 = 컨텍스트 분리(주), 병렬(부). | | 캐릭터 고유 Q/W/E/R | 그 에이전트의 역할/시스템 프롬프트 | ⚠ 스킬 아님. 캐릭터에 고정된 정체성. | | 아이템 / 소환사 주문(점멸·점화) | Skill (Agent Skill) | 누구나 장착·공유, 필요할 때 꺼내 끼움(progressive disclosure). | | 소환수(티버 등) | 스킬이 띄운 sub-agent | ⚠ 8편에서만. "스킬 호출이 격리된 하위 컨텍스트를 띄운다"의 비유. | | 매 턴 건네는 데이터 보따리 | context window(매 호출 전체 재전송) | LLM은 무상태. 캐시는 "다시 싸는 수고"만 줄임, 내용은 매번 감. | | 보따리가 무거우면 둔해짐 | context rot / lost in the middle | 절벽 아닌 그래디언트. | | 보따리 정리 | 컨텍스트 관리(2층: KV eviction + compaction) | GC가 아니라 손실 요약/제거. |

이 표가 시리즈의 헌법이다. 비유가 흔들리면 여기로 돌아온다.

2. 전체 구조 — 4막 11편 (게임의 진행처럼)

ACT 1. 게임의 규칙 (컨텍스트의 본질)

1강 · 이 캐릭터는 기억이 없다 — 무상태 & 보따리

  • 단일 메시지: "AI는 매 턴 context 보따리를 통째로 다시 건네야 움직인다. 어제의 대화도 사실은 매번 다시 싸서 넘긴 거다."
  • 캐릭터·스킬·소환수는 예고편(떡밥)만. (1강 over-scoping 방지)
  • 시뮬레이터: 빈 캐릭터에게 보따리를 싸서 던지면 행동 → 보따리 비우면 다시 백지.

2강 · 왜 모르면서 아는 척하는가 — 할루시네이션

  • 1강에서 모델은 "고정 함수가 매 턴 다음 토큰 확률분포를 뱉는다"고 배웠다. 핵심: 모델은 항상 그럴듯한 다음 토큰을 내놓는다 — "모른다"는 기본 상태가 없다.
  • ★ 한 줄: 모델은 '참(truth)'이 아니라 '그럴듯함(plausibility)'을 최적화한다. 학습 안 된 영역에선 그럴듯함과 참이 갈라지고, 그 틈이 할루시네이션.
  • ⚠ 정정 1: "가장 높은 확률"이 아니라 분포에서 샘플링(temperature/top-p). 온도↑ → 헛소리↑.
  • ⚠ 정정 2: "벡터에서 가까운 걸 찾는다"는 DB 조회가 아니다. 검색이 아니라 생성 — 학습된 매끄러운 함수가 항상 분포를 출력.
  • ★ 최신(OpenAI 2025, "Why Language Models Hallucinate"): 모델이 "모른다"고 안 하는 건 학습·평가가 추측에 보상을 주기 때문. 빈칸=0점, 찍기=가끔 정답 → 평균적으론 찍는 게 점수↑. 우리가 "자신 있게 찍어라"라고 가르친 셈.
  • 비유(게임 아님): ① 빈칸 못 내는 시험 학생(생일 모르면 "9/10" 찍기 = 1/365 vs "모름"=0점), ② 작화증(confabulation) — 환각이 아니라, 빈틈을 본인도 모르게 그럴듯하게 메우는 것. "거짓말이 아니라 작화한다."
  • 시뮬레이터: 같은 질문에 temperature 0 vs 높음 비교 / "모른다" 옵션을 막았을 때 vs 허용했을 때 정답률·헛소리율.

3강 · 보따리는 공짜가 아니다 — 어텐션 n² & KV 캐시

  • 어텐션은 n개 토큰에 n² 쌍 계산 → 늘리려면 알고리즘이 필요(FlashAttention, GQA/MQA, PagedAttention, RoPE 보간/YaRN).
  • KV 캐시: 재계산을 없애 토큰당 O(n)으로 떨어뜨리지만, 캐시 메모리가 새 병목이 됨.
  • 시뮬레이터: 토큰 수↑ → 연결선 n²로 폭증 + 비용 곡선. KV 캐시 on/off 비교.

4강 · 보따리가 무거우면 둔해진다 — context rot / lost in the middle

  • 시작·끝은 잘 보고 중간을 흘린다. Chroma 2025: 18개 모델 전부 저하. 절벽 아닌 그래디언트.
  • ⚠ "캐시하면 길어도 괜찮다"는 착각 분리: 캐시=비용/지연 해결, 품질은 별개.
  • 시뮬레이터: needle 위치(앞/중간/끝)별 회수 정확도 막대.

ACT 2. 보따리 관리 (컨텍스트 엔지니어링)

5강 · 보따리 청소에는 두 개의 층이 있다 — 컨텍스트의 'GC'

  • 추론 레이어: KV eviction / StreamingLLM(attention sink + 최근 윈도우, 중간 폐기).
  • 앱 레이어: compaction(요약 압축), tool result clearing(죽은 데이터 제거).
  • ★ 정정: 이건 자동 reachability GC가 아니라 손실 요약이다. 그래서 어렵다.
  • 시뮬레이터: 보따리 항목이 색으로 분류되어 비워지거나 압축되는 과정.

6강 · 버리지 말고 빼두기 — 결정 로그 & negative knowledge

  • structured note-taking(NOTES.md / memory tool), 결정 로그(ADR).
  • ★ 정정: "다 지우고 결정만"은 좋지만 실패한 시도와 이유(negative knowledge)도 일급 항목으로 남겨야 같은 삽질 방지.
  • 안정적 프리픽스(시스템 프롬프트·결정 로그)를 앞에 두면 prompt caching도 잘 먹고 lost-in-the-middle도 덜 탐.
  • 시뮬레이터: 보따리에서 결정·실패기록을 외부 노트로 빼고 다음 턴에 다시 부르는 흐름.

ACT 3. 캐릭터와 스킬 (에이전트 / 스킬)

7강 · 왜 캐릭터를 나누는가 — 서브에이전트 = 깨끗한 시야

  • 핵심 통찰: 나누는 1번 목적은 병렬이 아니라 컨텍스트 분리. 병렬은 보너스.
  • 서브는 수만 토큰 탐색해도 1,000~2,000토큰 요약만 들고 귀환 → 메인 보따리 오염 안 됨.
  • 시뮬레이터: 메인 보따리는 깨끗, 서브 보따리만 부풀었다가 요약만 반환.

8강 · 스킬은 아이템이다 — Agent Skill & progressive disclosure

  • 스킬 = 아이템/소환사 주문: 누구나 장착·공유, 동적 발견(메타데이터 상주 → SKILL.md 본문 → 하위 리소스 3단계 지연 로딩).
  • 범용으로 만들지 말고 프로젝트 맞춤으로. 스킬이 소환수를 부르기도(스킬→sub-agent, 애니의 티버).
  • 시뮬레이터: 아이템 상자에서 필요한 스킬만 꺼내 장착, 미사용 스킬은 보따리에 안 들어감.

9강 · 부대 편성의 기술 — orchestrator-worker & 병렬

  • 여기서 비로소 병렬 이득. 의존 없는 분해 = 병목 제거(서로 안 기다림).
  • Anthropic 멀티에이전트: 복잡 작업 단일 대비 90%+ 향상, 시간 최대 90%↓.
  • 시뮬레이터: 직렬(한 줄로 대기) vs 병렬(독립 분기) 처리 시간 비교.

ACT 4. 실전 루프 (자동화)

10강 · 죽으면 되살아나기 — 통합테스트 게이트 & 피드백 루프

  • 룰을 주입하기보다 통합테스트로 영향을 잡고, 깨지면 롤백 후 재개발.
  • ★ 핵심: "깨졌다"만 주지 말고 무엇을 왜 깼는지(영향 분석)를 다음 시도의 컨텍스트로 먹인다. 블라인드 재시도는 thrashing.
  • 2층 구조: 빠른 유닛테스트로 루프 + 통합테스트는 게이트, 시도마다 git 체크포인트.
  • ⚠ TDD 정정: 절차만 강요하면 작은 모델은 회귀 증가(TDAD). 핵심은 테스트-컨텍스트.
  • 시뮬레이터: 통합테스트 빨간불 → 롤백 → 실패 원인 카드가 다음 보따리에 삽입 → 재도전.

11강 · 보스전: 진짜 업무 자동화 — 원리 총동원

  • 1~9강 원리를 한 워크플로(예: Codex/Claude로 반복 업무 1개 자동화)에 전부 적용해 끝까지.
  • ⚠ 도구 데모는 날짜 박제 별도 트랙으로(아래 4절).

3. 한 편 고정 포맷 (10~13분)

  1. 0:00–1:00 훅 — 게임 상황으로 현상 제시 ("이 캐릭터가 왜 갑자기 멍청해졌을까?")
  2. 1:00–4:00 판서 — 원리 1개 손으로 전개 (메타코딩 정체성)
  3. 4:00–8:00 시뮬레이터 — 그 개념 1개만 (★ "개념 비유 — 실제 내부와 다름" 라벨 고정)
  4. 8:00–11:00 실전 한 스푼 — 실제 Claude/Codex에서 1번
  5. 11:00–끝 — 1줄 요약 + 다음 편(=다음 영입/다음 규칙) 떡밥

4. 제작 파이프라인 (번아웃·품질편차 방지)

  • 시뮬레이터 디자인 시스템 1개 고정 — 색·폰트·레이아웃 CSS 토큰 템플릿화 → 회차 톤 통일, 매 편 Claude에 같은 톤 지시.
  • 게임 자산 재사용 — 캐릭터/보따리/아이템 UI를 한 번 만들어 전 편 공유 → 제작비↓, 시리즈 일관성↑.
  • 비유 라벨 의무화 — 오개념 방지.
  • 원리편 evergreen / 도구편 날짜 박제 — 도구편 제목·설명에 "(2026-05 기준)" 명시.
  • 회당 개념 1개 상한 — 시뮬레이터 1개 = 개념 1개.

5. 1강 상세 설계 — "이 캐릭터는 기억이 없다" (★ 게임 데뷔편)

5-1. 1강이 가르치는 단 하나

"AI 캐릭터는 무상태(stateless)다. 매 턴, 지금까지의 모든 것을 context 보따리에 담아 통째로 다시 건네야만 움직인다."

이것 하나만. 캐릭터 분리·스킬·소환은 절대 1강에서 설명하지 않는다(예고만).

5-1.5. 왜 기억이 없어야만 하는가 (★ 1강의 진짜 핵심)

"기억이 없다"는 현상만 가르치면 약하다. "없어야만 하는 이유" 를 가르쳐야 1강이 깊어진다. 이유는 네 겹.

  1. 모델은 고정된 함수다. 학습 끝난 가중치는 얼어 있다. 추론은 출력 = f(고정된 뇌, 입력 토큰). 대화해도 가중치는 안 바뀐다 → 모델 안에 기억을 저장할 공간 자체가 없다.
  2. 그래야 수백만 명을 동시 서비스할 수 있다(진짜 'must'). 상태가 모델 인스턴스에 남으면 다음 요청을 다른 GPU로 못 보낸다. 무상태라서 어떤 복제본이든 "보따리만 받으면" 응대 가능. 웹 서버를 stateless(REST)하게 짜는 것과 같은 이유 — 확장성.
  3. 그래야 재현·디버깅·감사가 된다. 출력이 입력 보따리에만 의존 → 같은 보따리 = 같은 행동. 숨은 내부 기억이 있으면 추적 불가. 무상태라서 출력에 영향 주는 모든 게 보따리 안에 보인다. → "보따리에 없으면 일어나지 않은 일", 그래서 컨텍스트 설계가 전부.
  4. 그래야 통제권이 당신에게 온다. 상태가 밖에 명시적이라 매 턴 편집·삭제·리셋 가능. 내부 기억이면 못 지운다.

1강 thesis: "캐릭터의 건망증은 극복할 결함이 아니라 당신에게 통제권을 주는 설계다. 얘가 스스로 아무것도 기억 못 하니까, 얘가 아는 건 전부 당신이 보따리에 넣은 것뿐이다 — 그래서 플레이어인 당신이 중요하다. 이 시리즈는 그 보따리를 싸는 법이다."

모델 vs 앱(하네스) 구분: "ChatGPT는 어떻게 기억하지?"의 답 = 기억하는 건 모델이 아니라 앱이다. 앱이 대화 기록을 DB에 저장했다가 매 호출 보따리에 다시 욱여넣는다. 모델 = 건망증 환자, 기억지기 = 앱.

5-2. 흐름 (12분 안)

  1. 훅(0:00) — 화면 속 캐릭터에게 "방금 한 말 기억해?" → 캐릭터 백지. "얘는 금붕어인가? 아니, 원래 기억이 없다."
  2. 판서(1:00) — 사람은 머릿속에 상태가 남지만, LLM은 함수다. 출력 = f(보따리). 보따리 밖의 것은 존재하지 않는다.
  3. 반전(3:30) — "그럼 ChatGPT는 어떻게 이전 대화를 기억하지?" → 사실 기억하는 게 아니라, 매 호출 이전 대화 전체를 보따리에 다시 싸서 넘기는 것.
  4. 시뮬레이터(4:30) — 빈 캐릭터 + 보따리 UI. 보따리에 메모를 넣고 "턴 실행" → 캐릭터가 그 내용 기반 행동. 보따리 비우고 다시 실행 → 백지. (보따리 = 캐릭터의 전 우주)
  5. 떡밥(8:30) — "근데 얘는 모르면 솔직히 모른다고 할까? 아니, 그럴듯하게 지어낸다" → 2강(할루시네이션) 예고. "보따리를 무한정 키울 순 있을까?" → 3강(어텐션·KV) 예고.
  6. 시즌 예고(10:30) — "다음 시즌엔 캐릭터를 여러 명 부리고(7강), 아이템을 끼우고(8강), 소환수도 부른다(8강)." 떡밥만.
  7. 1줄 요약 — "기억하지 못하는 게 버그가 아니라 사양이다. 그래서 우리가 보따리를 설계한다."

5-3. 1강에서 절대 하지 말 것

  • 캐릭터 분리/병렬/스킬/소환을 "설명"하기 (떡밥은 OK, 설명은 NO).

7. 사실 검증 체크리스트 (대본 부착용)

  • [ ] "토큰 못 늘린다" → "늘리려면 알고리즘 필요 + 수확 체감".
  • [ ] 할루시네이션을 "검색 실패"로 설명 안 했는가 → "항상 그럴듯한 토큰을 뱉는 함수(생성) + 추측 보상". '참'이 아니라 '그럴듯함' 최적화, 작화증.
  • [ ] "가장 높은 확률"이라 단정 안 했는가 → "분포에서 샘플링(temperature)".
  • [ ] context rot을 "절벽"이라 안 했는가 → "그래디언트".
  • [ ] 캐시가 품질을 고친다고 안 했는가 → "캐시=비용/지연, 품질은 컨텍스트 엔지니어링".

  • [ ] 서브에이전트 목적을 "컨텍스트 분리 1번, 병렬 보너스"로 말했는가.
